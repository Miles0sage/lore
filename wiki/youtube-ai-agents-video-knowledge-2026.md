### 3. Claude Code's Agent Teams Are Insane
**Video:** https://youtube.com/watch?v=-1K_ZWDKpU0  
**Channel:** Cole Medin | **Views:** 107K | **Duration:** 20:16  
**Why selected:** Most detailed practical breakdown of Claude Code's Agent Teams feature — setup, architecture, sub-agent vs. agent team tradeoffs, contract-first spawning pattern.

**Full transcript:**

> Take a look at this. I have four instances of Claude code working at the exact same time together to perform a code review on my codebase. And this is all thanks to the new agents teams feature that Anthropic has built into Claude Code. And man, let me tell you, it really does look like I'm peering into the future of agentic engineering when I am using it. So, we have our primary lead agent on the left-hand side. And in real time, I watched it spin up each one of these T-Mux terminals to create these agents to collaborate on the same task.
>
> Now, people have been doing this sort of split pane multi-tmux terminal sort of setup for a while now. So it's not really new, but there are a couple of things that make this super novel. The first is that our primary agent actually decided the team to form based on the request that I gave it. And the other part of this that makes it so powerful is that under the hood, each one of these agents is working on the exact same task list together. So this goes way beyond sub agents. These agents actually talk to each other like, "Oh, let me complete this before you work on this." They have that kind of communication that makes it so that we can take this idea of parallel agents a lot further.
>
> **Anthropic used 16 agents running together with this new agent teams feature to build an entire C compiler. Building a compiler from scratch is not easy. If you were to hire a dev team to do this, it would probably be hundreds of thousands of dollars. But they were able to do it with only $20,000 in API costs.**
>
> [Setup] The first thing you have to do because this is an experimental feature — you have to enable it. You can either set this environment variable on your computer or in just a terminal session or you can add this to your settings.json. So this is one of the config files for Claude Code. You've probably worked with this before because it's where you set things like your MCP servers and your hooks. And so we can set this at either the global .claude level or in the .claude project directory. You can enable agent teams just for specific projects if you want. The other thing that you have to set up if you want that split pane mode where you can see all the terminals at the exact same time is you need to install either T-Mux or iTerm 2.
>
> **Sub Agents vs. Agent Teams:**
>
> The primary idea with sub agents is context isolation. We want some way to be able to dish out a request that could take tens or even hundreds of thousands of tokens, but all we need back is a summary. So our primary agent knows generally what happened, but it doesn't have to be polluted by the context of the entire task. This is important because context is the most precious resource when you're using an AI coding assistant. But this context isolation has downsides because there is no coordination between sub agents and the entire process is just a black box — because it is only the summary, the final output, that is given back to our primary agent.
>
> Sub agents are generally used for focused tasks — usually something like research — because all we care about is the result, that summary at the end of the research. If we're doing something like coding and we have a sub agent actually write code, then we don't have any idea into the process of the sub agent and so the main agent loses a lot of context as to what was actually implemented.
>
> With agent teams, we still have a primary agent spinning off these subprocesses, but the difference here is they're actually talking to each other. So they have this shared task list. They're updating each other on their progress, communicating to the main agent as well. We have true peer-to-peer coordination. And this is so powerful for implementation — for example, our backend agent might change something in an API endpoint where it would have to tell the frontend agent, "Hey, I changed this API. Make sure you update the frontend component that uses the API as well." When we had sub agents doing that kind of implementation, those kinds of things would break all the time because they're not talking to each other.
>
> A rough estimate: agent teams use two to four times the token usage compared to just using Claude Code by itself or using sub agents. So generally: **use sub agents for research, use agent teams for implementation.** A lot of times when you're working with a single conversation of Claude Code, you might start with sub agent research like analyzing the codebase, then create that task list and spin up the agent team to knock out the plan.
>
> **Contract-First Spawning pattern:** Don't do everything in parallel. Set the stage up front for work that has to be done before you can kick off all agents — like "here's our database schema" — then send the agents to work in parallel. This has gotten very reliable results vs. just telling Claude "spin up an agent team to do XYZ" without additional instructions. Database agent first, then backend agent, then frontend agent — each sends its "contract" before downstream agents begin.
>
> **Problems with Agent Teams (current limitations):**
> 1. You have to be very specific with Claude about team composition — vague prompts cause hallucinated or misformed teams
> 2. Agents can't truly run in parallel when there are ordering dependencies (e.g., backend needs database schema to exist first)
> 3. Limited visibility into actual agent collaboration — you have to trust the task list is being managed correctly
> 4. Token-heavy — 2-4x compared to single-agent usage
>
> **Recommendation template (the /build-with-agent-team skill):**
> - Give Claude a pre-made plan (from sub-agent research)
> - Instructions tell it: determine the optimal team composition, manage terminals, use contract-first spawning
> - Can define number of agents or let Claude decide dynamically
> - Works for both new projects and features in existing codebases

## Key Insights Synthesized

### Agent Teams vs. Sub-Agents (Claude Code)

| Dimension | Sub-Agents | Agent Teams |
|-----------|-----------|-------------|
| Coordination | None (black box) | Peer-to-peer, shared task list |
| Context sharing | Summary only | Full collaboration |
| Token cost | Low | 2-4x higher |
| Best for | Research, isolated tasks | Implementation, interdependent work |
| Visibility | Opaque | Moderate (tmux panes) |
| Ordering | Independent | Contract-first spawning for deps |

### Agentic Workflow Building Blocks (Stanford CS230)

1. **Prompting** — zero-shot, few-shot, chain-of-thought, self-consistency
2. **RAG** — retrieve external knowledge before generation; reduces hallucination
3. **Fine-tuning** — avoid unless prompting + RAG is insufficient; expensive and brittle
4. **Agentic loops** — LLM decides actions, executes tools, observes results, repeats
5. **Multi-agent** — specialized agents with defined roles, supervisor coordinates
6. **Evals** — measure agent performance before deploying; treat like unit tests

### Advanced Patterns from Nick Saraev's Course

- **Self-modifying agent instructions**: agents rewrite their own system prompts to reduce recurring errors
- **MCP orchestration**: register multiple AI models (Claude, Gemini, Codex) as MCP servers; manage from one thread
- **Stochastic consensus**: spawn N agents with same prompt, aggregate outputs statistically to improve quality
- **Verification loops**: agents peer-review each other's outputs in real time
- **Video-to-action pipelines**: agents learn from YouTube video transcripts rather than just text
- **Prompt contracts**: formal interface definitions between agents (what inputs/outputs to expect)
- **Reverse prompting**: have the LLM generate the optimal prompt for a given task before executing

### Framework Signals from AutoGen/Microsoft Search

- AutoGen 0.4 is in maintenance mode; Microsoft has shifted to the broader "Agent Framework"
- LangGraph leads for stateful, complex workflows
- CrewAI leads for rapid role-based prototyping
- For 2026: LangGraph > CrewAI > OpenAI Agents SDK for most production use cases
- AutoGen vs CrewAI vs LangGraph is now a settled debate — LangGraph wins production, CrewAI wins speed-to-demo
