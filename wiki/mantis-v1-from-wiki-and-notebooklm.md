# MANTIS v1 From Wiki and NotebookLM

*Sources: ai-agents wiki + NotebookLM synthesis | Date: 2026-04-04*

---

## What The Wiki Work Was Actually Doing

The wiki was not the product. It was the research and decision layer used to answer a narrower question:

**What should MANTIS actually be?**

The process was:

1. collect and clean Claude Code ecosystem sources
2. distill them into linked wiki pages
3. feed the strongest ones into NotebookLM
4. ask what is durable, what is hype, and what is still missing
5. use that to define the MANTIS cut

So the wiki work existed to reduce product ambiguity.

---

## What The Notebook Converged On

The strongest recurring extension surfaces around Claude Code were:

- hooks
- skills
- MCP toolchains
- observability / cost tracking
- semantic code intelligence
- CI / automation

The strongest product gaps still open were:

- persistent session state and resume
- precise code editing with semantic awareness
- native cost visibility
- real verification loops
- stronger CI/CD integration

The most defensible advantages for an open competing product were:

1. semantic code understanding
2. better verification
3. better session continuity
4. better CI automation
5. better self-hosting/privacy
6. lower cost

That pushes MANTIS toward a narrower coding-agent core rather than a sprawling agent platform.

---

## What MANTIS Should Be

MANTIS should not be “all of AI Factory.”

The cleaner split is:

- **AI Factory** as the source engine and parts bin
- **MANTIS** as the stripped product shell

That means the right move is not inventing a new product. It is finishing MANTIS by narrowing it.

### Recommended v1 definition

MANTIS v1 should be:

> An open, self-hosted coding agent that remembers project state, edits code precisely, verifies its work, and works with any model.

---

## What To Focus On

### Build first

- persistent session state and resume
- semantic code understanding
- precise edit application
- verification loop
- permission gates
- correct cost visibility

### Build second

- GitHub / CI automation
- model routing
- shell / web UX polish

### Park for later

- giant swarm orchestration
- proactive daemon mode
- oversized dashboard surface
- broad business-agent features
- every experimental subsystem from AI Factory

---

## Why This Is Worth Building

This product is worth building if it helps users do a few things materially better:

- resume real coding work without context loss
- understand large codebases semantically
- make safer multi-file changes
- verify before claiming success
- keep permission and cost control local
- run on the models and infrastructure they choose

That is a real wedge.

What is not a wedge:

- having agents for the sake of agents
- having swarms because competitors do
- cloning Claude Code superficially

The actual wedge is:

**memory + verification + semantic tooling + control**

---

## Recommended Build Order

1. wire session persistence
2. wire semantic code tooling
3. wire precise edit applicator
4. wire the verification loop fully
5. wire permission checks
6. fix cost accounting
7. tighten the CLI / web path around that core
8. add CI integration later
9. add subagents only after the core is solid
